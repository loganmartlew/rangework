# Brand Asset Generation — Implementation Plan

Single-source the Rangework brand mark. Today the same geometry (three perspective bands + a
tee-shaped rod) is hand-replicated across ~13 files in three formats. This plan derives every
variant from one role-annotated base SVG + a declarative manifest, generated by extending
`@rangework/design` into a small design-asset build pipeline.

See [ADR 0001](../../docs/adr/0001-brand-asset-generation-pipeline.md) for the decision record.

## Outcomes

- One base SVG + one manifest are the only hand-edited brand sources (plus two documented
  exceptions: `ic_notification.xml` and the `mipmap-anydpi` adaptive-icon defs).
- Every variant colour resolves from `color.tokens.json` — zero hardcoded hex in the pipeline.
- All generated assets are gitignored, consumed in place by the Android build and served to the
  site by a Vite plugin. Only the Play Store icon is committed.
- `apps/site/` carries **zero** committed brand assets.

## 1. Package architecture — the design package becomes a generator pipeline

Refactor `packages/design/scripts/build.mjs` from a Style-Dictionary wrapper into an
orchestrator over ordered **generators**.

### 1.1 Generator contract

```ts
interface Generator {
  name: string;
  build(ctx: BuildContext): Promise<void>;
}

interface BuildContext {
  packageDir: string;   // absolute package root
  outDir: string;       // generated/  (gitignored)
  distDir: string;      // dist/web/   (gitignored, existing web token outputs live here)
  tokens: ResolvedTokens; // color.tokens.json + typography, parsed ONCE
  log: (msg: string) => void; // namespaced per-generator
}
```

- `tokens` is resolved **once** by the orchestrator and shared with every generator. The brand
  generator reads `ctx.tokens` (e.g. `ctx.tokens.color.brandRod.light`) — it never re-parses
  JSON or hardcodes hex. This is the single-source mechanism.
- Plain ordered array, run sequentially. **No dependency graph** — the shared pre-resolved
  `ctx.tokens` removes the only ordering coupling. Add a graph only if a future generator needs
  another generator's *file* outputs (YAGNI today).
- "Generator" is the canonical term (not "plugin" — there is no dynamic/third-party discovery).

### 1.2 Config registry

`packages/design/design.config.mjs`:

```js
import { tokensGenerator } from './generators/tokens/index.mjs';
import { brandGenerator } from './generators/brand/index.mjs';

export default { generators: [tokensGenerator(), brandGenerator()] };
```

Adding a capability = new generator module + one line here.

### 1.3 Orchestrator (`scripts/build.mjs`)

1. `rm -rf dist generated`.
2. Resolve tokens once (reuse Style Dictionary's parse, or parse `tokens/**/*.json` directly).
3. Build `BuildContext`.
4. `for (const g of config.generators) await g.build(ctx)` with namespaced logging.

### 1.4 `tokensGenerator`

Wraps the **existing** `style-dictionary.config.mjs` unchanged. It is now one generator among
peers; current outputs stay byte-identical (`dist/web/*.css`, `generated/android/{kotlin,res}`).

## 2. The base SVG and shared geometry model

### 2.1 Base file — `packages/design/brand/mark.base.svg`

- `viewBox="0 0 108 108"`, single `<g transform="translate(0,4)">`, five `<path>` elements.
- Each path carries:
  - `data-role` — `band` (×3) or `rod` (×2).
  - intrinsic `fill-opacity` — `1`, `0.48`, `0.20` for the ground/mid/far bands; `1` for rod
    shaft and cap. **Opacity is design/geometry, identical across variants → it lives in the
    base.** Colour does **not**.
- No fill colours (omit `fill`, or use a neutral placeholder stripped at parse).
- Path `d` values are the canonical geometry, lifted verbatim from the current
  `ic_launcher_foreground.xml` (108-grid), which all Class 1 files already mirror.

### 2.2 Parse step (shared helper)

Use `svgson` (not regex) to parse the base into:

```ts
interface MarkModel {
  viewBox: [number, number, number, number];
  transform: string;            // "translate(0,4)"
  paths: { role: 'band' | 'rod'; opacity: number; d: string }[];
}
```

Generators **emit each target format from `MarkModel`** — they never text-transform SVG into
another format. This is what lets Android emit `@color/...` token references rather than baked
hex.

## 3. Variant manifest

`packages/design/brand/manifest.mjs` — declarative list of every generated output.

### 3.1 Colour expression

Each role maps to a colour treatment that resolves from tokens:

```
role -> {
  mode: 'currentColor'              // web mono
       | 'token-responsive'         // Android @color/... (flips light/dark at runtime)
       | 'token-fixed',             // pin token to a specific theme value
  token?: string,                   // e.g. 'brandRod' (when not currentColor)
  theme?: 'light' | 'dark',         // required when mode = 'token-fixed'
}
```

### 3.2 Variant shape

```
{
  name,
  target: 'web-svg' | 'android-vector' | 'raster',
  colors: { band: <colorExpr>, rod: <colorExpr> },
  background?: { token, theme, shape: 'square' | 'rounded', radius? },
  crop?: [minX, minY, w, h],        // logo variant only
  outputs: [ '<path under generated/ or dist/>' ],
  raster?: { size, format: 'ico' | 'png', sizes?: number[], bitDepth? },
}
```

## 5. Generators

### 5.1 `brandGenerator` — web SVG + Android vector

- Reads `mark.base.svg` → `MarkModel`, reads `manifest.mjs`.
- **web-svg emitter:** writes `<path fill="…">` (resolved hex) or `currentColor`; wraps optional
  background `<rect>`; applies `crop` by rewriting `viewBox`.
- **android-vector emitter:** writes `<vector>` with `<group android:translateY="4">` and
  `<path android:fillColor="@color/rangework_mark_band" android:fillAlpha="0.48" …>`; emits
  `android:tint="@color/rangework_mark_band"` for the mono variant; emits `<inset>` / `<color>`
  wrappers (Class 2) directly from the manifest.

### 5.2 raster emitter

- Renders a composed variant SVG via `@resvg/resvg-js`.
- `favicon.ico`: render at 16/32/48 → pack with `png-to-ico`.
- `play-icon`: render at 512×512 → PNG (32-bit RGBA, no rounding/shadow — Google Play applies
  its own mask).
- New dev deps on `@rangework/design`: `svgson`, `@resvg/resvg-js`, `png-to-ico`.
